## Overview

When two words could mean the same thing, this page says which one we use and what it means.

## Terms

**Action** тАФ The enforcement outcome of a verdict. One of `allow`, `warn`, `block`, `quarantine`. Distinct from severity.

**Admission** тАФ The install-time gate that decides whether a skill/MCP/plugin/tool is allowed into the agent runtime. Implemented by `admission.rego`.

**Admission override** тАФ An operator decision to install an artifact despite a `block` verdict, recorded in the audit store with a required `reason`.

**AIBOM** тАФ AI Bill of Materials. CycloneDX-shaped inventory of models, datasets, skills, and plugins installed on the host.

**Audit bridge** тАФ The translator that maps persisted audit events into `gatewaylog.Event` lifecycle rows for `gateway.jsonl`.

**Audit store** тАФ The local SQLite database of every decision and verdict. Authoritative; all other sinks are copies.

**Bifrost** тАФ The provider-routing layer used by the Go gateway to call configured LLM providers.

**ClawShield** тАФ The binary/archive/image scanner.

**CodeGuard** тАФ The generated-code scanner.

**Correlation ID** тАФ A string carried end-to-end across a single agent turn so every emitted event can be joined back into one story. Format: `req_01HX...` (ULID-like).

**Direction** тАФ `prompt` | `completion` | `tool_call`. The role of the content being inspected.

**Drift** тАФ Content of an already-approved artifact changed since approval. Detected by snapshot-diff in the watcher.

**Enforcement** тАФ The physical-layer action that follows an admission decision (quarantine, restore, skip). Distinct from admission which is the decision.

**Event** тАФ A structured record of something happening. Produced by every subsystem; persisted in the audit store; forwarded to sinks.

**Event filter** тАФ The sink-level `min_severity` and `actions` filters that decide whether an audit event is delivered.

**Finding** тАФ A single detection emitted by a scanner or rule. Identified by a prefixed string (`secret:aws-key`, `injection:shell`, `pii:email`). Multiple findings compose a verdict.

**Guardrail** тАФ The LLM traffic inspector: the proxy + scanner pipeline that inspects prompts, completions, tool calls.

**Judge** тАФ The LLM-based adjudicator used in `regex_judge` and `judge_first` detection strategies.

**LLM judge client** тАФ The Go client under `internal/gateway/llm_judge.go` that calls the judge, enforces timeouts, and caches verdicts.

**Master key** тАФ The default API bearer token minted at install time.

**MCP** тАФ Model Context Protocol. The standard for agent tool providers.

**Mode** тАФ `observe` | `action` | `disabled`. Controls whether the guardrail logs or enforces.

**Notification queue** тАФ The in-process queue used by the guardrail path to inject pending enforcement notifications into model-facing system messages.

**OpenShell** тАФ The CLI that invokes shell subprocesses on behalf of agents. Sandbox-wrapped by DefenseClaw.

**Pack version** тАФ A hash of the compiled rule pack + suppressions snapshot. Used to version-scope the verdict cache.

**Policy snapshot** тАФ An immutable compile of Rego modules + YAML rules + `data.json` overlays. Swapped atomically on reload.

**Profile** тАФ A named rule-pack directory: `default`, `strict`, `permissive`, or custom.

**Quarantine** тАФ The physical move of a blocked artifact to `~/.defenseclaw/quarantine/`.

**Redaction** тАФ Scrubbing sensitive bytes from events before they leave the gateway.

**Scanner** тАФ Any component that produces a `ScanVerdict`.

**Scanner mode** тАФ `local` | `remote` | `both`. Controls where the scanner runs.

**Sensitive tool** тАФ An MCP tool marked in `sensitive-tools.yaml` for tighter result inspection.

**Sidecar** тАФ The Go process (`defenseclaw-gateway`) that owns policy, audit, and the guardrail proxy.

**Sink** тАФ An `audit_sinks[]` destination for audit events, currently `splunk_hec`, `otlp_logs`, or `http_jsonl`.

**SSRF** тАФ Server-Side Request Forgery. The firewall's SSRF guard mitigates this.

**Strategy** тАФ `regex_only` | `regex_judge` | `judge_first`. Controls how the guardrail decides.

**Suppression** тАФ A declarative rule that drops or strips content to eliminate false positives.

**TUI** тАФ The terminal UI (`defenseclaw tui`) that surfaces everything the CLI can show, interactively.

**Verdict** тАФ The structured output of a scan. `severity`, `findings`, `action`, `reason`, `confidence`.

**Verdict cache** тАФ In-process cache for guardrail verdicts, keyed by content/direction/model/kind in the current source.

**Watcher** тАФ The filesystem observer that triggers scanners on install/change.
