> "I want to observe Claude Code and block recursive force deletion against critical system paths."

That sentence is the original DefenseClaw user story. Here is the entire flow, end to end.

## Outcome

By the end of this story, you'll have:

- Claude Code wired to DefenseClaw via [native hooks + OTel](/docs/connectors/claudecode).
- A week of audit data in `~/.defenseclaw/audit.db` proving what your agent actually does.
- An `action` mode policy that blocks recursive force deletion under protected
  root prefixes with a CRITICAL finding.
- An audit row showing the block, with the rule reference for the next reviewer.

<Steps>

<Step>

### Wire Claude Code (observability only)

```bash
defenseclaw setup claude-code
```

This is the [observability-only alias](/docs/connectors/claudecode). Claude Code talks directly to its native upstream. DefenseClaw inspects via PreToolUse / PostToolUse / UserPromptSubmit / Stop / PermissionRequest hooks plus the native OTel exporter.

</Step>

<Step>

### Watch the audit log fill up

Open the live dashboard:

```bash
defenseclaw tui
```

Or inspect a scriptable snapshot from mandatory SQLite event history:

```bash
defenseclaw-gateway audit export --limit 200 --output - \
  | jq 'select(.connector == "claudecode")'
```

Open Claude Code and use it normally. Every prompt, tool call, and response shows up. Run this for a week — you want a real picture of your agent's behaviour before the first block decision.

</Step>

<Step>

### Inspect the rule pack

```bash
ls ~/.defenseclaw/policies/guardrail/default/
sed -n '80,125p' ~/.defenseclaw/policies/guardrail/default/rules/commands.yaml
```

The `CMD-RM-RF` rule is CRITICAL and matches `/` plus protected top-level
prefixes such as `/etc`, `/var`, `/home`, `/usr`, and `/opt`. It deliberately
does not flag every `rm -rf` command or arbitrary home subdirectory. The three
bundled packs currently carry the same command-rule boundary; their policy
thresholds and other configuration differ.

</Step>

<Step>

### Promote to action mode with HITL

```bash
defenseclaw setup guardrail \
  --connector claudecode \
  --mode action \
  --rule-pack default \
  --human-approval \
  --hilt-min-severity high \
  --restart
```

Three flags matter here:

- `--mode action` — CRITICAL findings now block.
- `--human-approval` — HIGH findings become confirmation requests on ask-capable events.
- `--hilt-min-severity high` — MEDIUM and LOW still log; only HIGH and above prompt.

Claude Code supports native PreToolUse ask, so the approval prompt surfaces inside the agent UI itself.

</Step>

<Step>

### Trigger the rule

Create an empty target so the exercise remains safe even if the connector is
not enforcing:

```bash
mkdir -p /var/tmp/defenseclaw-demo-empty
```

Then ask Claude Code:

> Please run `rm -rf /var/tmp/defenseclaw-demo-empty`.

The PreToolUse hook fires, the gateway scores the command, the default rule pack
returns `CMD-RM-RF` at CRITICAL, and Claude Code receives a deny decision before
the shell runs.

</Step>

<Step>

### Confirm in the audit log

```bash
defenseclaw alerts --limit 10
# or inspect the mandatory SQLite history filtered by severity:
defenseclaw-gateway audit export --limit 200 --output - \
  | jq 'select(.connector == "claudecode" and (.severity == "HIGH" or .severity == "CRITICAL"))'
```

```
2026-05-08T14:02:11Z  CRITICAL  CMD-RM-RF  blocked
  command:    rm -rf /var/tmp/defenseclaw-demo-empty
  reason:     Recursive force delete from critical root path
  rule:       policies/guardrail/default/rules/commands.yaml
  connector:  claudecode
  hook:       PreToolUse
  decision:   block
```

</Step>

</Steps>

## Variations

<Accordions>

<Accordion title="What if I want to allow it on a specific path?">

Use a less destructive command when possible. If your organization truly needs
a different boundary, copy the pack, edit `rules/commands.yaml`, validate the
custom pack, and select it with `defenseclaw setup guardrail --rule-pack-dir /absolute/path/to/pack --restart`. The current rule schema does not provide a
per-path allow-list field, and CRITICAL findings cannot be approved through
HITL.

</Accordion>

<Accordion title="What if I'm running an autonomous agent with no operator?">

Drop HITL (`--no-human-approval`) and rely on `--mode action`. In the default balanced profile, CRITICAL still blocks unconditionally while HIGH alerts instead of pausing. Use a strict or custom pack if HIGH must block, and configure a webhook sink so the audit row reaches your on-call channel.

</Accordion>

<Accordion title="What if I want the same protection on Codex / Cursor / Copilot CLI?">

`defenseclaw setup codex` (or `cursor`, `copilot`). The rule pack is connector-agnostic; the hook surface differs. See the [Capability Matrix](/docs/capability-matrix) for the per-connector hook events.

</Accordion>

<Accordion title="What if the gateway is unreachable when the hook fires?">

Transport failures are fail-open by default even though new installs use response-layer `hook_fail_mode: closed`. Set `DEFENSECLAW_STRICT_AVAILABILITY=1` to make supported hooks fail closed when the gateway is unreachable. Claude Code supports that path; check the [Capability Matrix](/docs/capability-matrix) for other connectors.

</Accordion>

</Accordions>
