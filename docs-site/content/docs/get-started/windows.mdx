DefenseClaw's certified Windows release runs directly on native Windows x64.
Certified agents send lifecycle events to the local DefenseClaw gateway through
their Windows hook surfaces. Observe mode records decisions; action mode can
return blocking or approval decisions before supported actions run.

<Callout>
This is a **native Windows x64** integration. **WSL is unsupported.**
`DefenseClawSetup-x64.exe` is a native Windows installer executable. The
installed CLI and TUI keep using DefenseClaw's audited Python application, run
from an embedded managed Python runtime installed with the product.
</Callout>

## Connector support

| Agent | Setup name | Release status on native Windows x64 | Important detail |
| --- | --- | --- | --- |
| Codex CLI | `codex` | certified | Native Codex hooks. |
| Claude Code | `claudecode` | certified | DefenseClaw setup does not require Git; follow the agent's own requirements. |
| Cursor, Windsurf, Gemini CLI, Copilot CLI, Antigravity, OpenCode, Hermes | - | not certified | Separate Windows certification is required. |

OpenHands, OmniGent, OpenClaw, and ZeptoClaw are not supported on native
Windows. OpenHands is WSL-only; OmniGent has no supported native terminal and
sandbox path; and the OpenClaw and ZeptoClaw connectors require a local proxy
that DefenseClaw does not host on Windows. Unsupported connectors are omitted
from Windows pickers and rejected by scripted setup.

## Requirements

- Native Windows x64 (`amd64`). Windows ARM64 requires separate certification; x86 is unsupported.
- A supported agent installed natively for the same Windows user.
- No separate Python, `uv`, Go, Node.js, Git, or PowerShell install command is required for the native setup path.

DefenseClaw installs per user by default under
`%LOCALAPPDATA%\Programs\DefenseClaw`. Administrator rights are not required.
Runtime state, configuration, connector backups, and audit history remain under
`%USERPROFILE%\.defenseclaw`.

## Install and initialize

<Steps>

<Step>

### Download the setup executable

Download `DefenseClawSetup-x64.exe` from the GitHub release page. Production
release artifacts are Authenticode signed by the DefenseClaw publisher and have
matching release checksums. Local or pull-request artifacts may be unsigned and
must be treated as test builds.

Double-click the file for the graphical setup wizard, or run a silent per-user
install:

```powershell
.\DefenseClawSetup-x64.exe /quiet /norestart INSTALLSCOPE=user CONNECTOR=codex MODE=observe STARTGATEWAY=1
```

Use `CONNECTOR=claudecode` for Claude Code, or `CONNECTOR=none` to configure
later. The setup executable contains the gateway, native hook launcher, Python
wheel, pinned Python runtime, and required Python packages, so installation does
not need network access after the file has been downloaded.

The finish screen directs you to `defenseclaw init` and `defenseclaw tui`. It
can open a terminal in the installed command directory, but it does not install
or claim a separate native DefenseClaw GUI application.

</Step>

<Step>

### Configure one or more agents

If you installed with `CONNECTOR=none`, run the wizard after setup:

```powershell
defenseclaw init
```

The wizard pre-selects detected hook connectors in observe mode and asks which
subset, if any, should use action mode. For one explicit connector:

```powershell
defenseclaw init --connector codex
```

</Step>

<Step>

### Verify the gateway and hook launcher

Open a new PowerShell or cmd window so the updated user `PATH` is loaded, then
run:

```powershell
Get-Command defenseclaw
Get-Command defenseclaw-gateway
Get-Command defenseclaw-hook

defenseclaw-gateway status
defenseclaw status
defenseclaw doctor
```

If initialization did not start the gateway, run `defenseclaw-gateway start`
and repeat `defenseclaw doctor`.

</Step>

<Step>

### Drive the native agent

Restart the configured agent so it reloads its hook configuration, then trigger
a prompt or tool call. Inspect the resulting decisions:

```powershell
defenseclaw tui
defenseclaw alerts
```

</Step>

</Steps>

## Repair, upgrade, and uninstall

Repair reinstalls the exact packaged files and keeps user configuration:

```powershell
.\DefenseClawSetup-x64.exe /repair /quiet /norestart INSTALLSCOPE=user
```

For normal upgrades, use the CLI. Native Windows installs download the setup
executable, verify the signed upgrade manifest, SHA-256 checksum, and
Authenticode publisher, then hand off to the setup executable so the running CLI
does not overwrite itself:

```powershell
defenseclaw upgrade --yes
```

Administrators can instead deploy the release installer through their managed
software channel. Managed update policy is respected so the self-updater does
not fight Intune or SCCM rollout. This setup executable is per-user only and
rejects `INSTALLSCOPE=machine`; this release path does not publish an MSI.

Uninstall removes product-owned application files, the installer cache,
Installed Apps registration, and the setup-owned user `PATH` entry. User data is preserved by
default:

```powershell
.\DefenseClawSetup-x64.exe /uninstall /quiet
```

Delete user data only when you explicitly want to remove configuration, audit
history, and connector backups:

```powershell
.\DefenseClawSetup-x64.exe /uninstall /quiet DELETEUSERDATA=1
```

## Observe, action, and multiple connectors

Observe mode is the safe starting point: findings are recorded without
blocking. Promote a connector to action only after reviewing the observed
traffic and confirming its installed version maps to a known
[hook contract](/docs/connectors/compatibility).

```powershell
# Add or refresh a connector in observe mode
defenseclaw setup codex --mode observe

# Enable native blocking for this connector
defenseclaw setup codex --mode action

# Remove only this connector's DefenseClaw wiring
defenseclaw setup remove codex
```

One gateway can protect both certified native hook connectors. For scripted
setup, repeat `--connector`:

```powershell
defenseclaw setup `
  --connector codex `
  --connector claudecode `
  --mode observe `
  --restart `
  --yes
```

Connector setup updates each agent's documented user-level configuration and
keeps the backups needed for targeted removal. A native agent and a copy of the
same agent inside WSL are separate installations with different executables,
home directories, and hook files; do not mix their configurations.

## Windows file layout

| Location | Purpose |
| --- | --- |
| `%LOCALAPPDATA%\Programs\DefenseClaw\bin\defenseclaw.exe` | Stable native command launcher for the embedded Python CLI/TUI. |
| `%LOCALAPPDATA%\Programs\DefenseClaw\bin\defenseclaw-gateway.exe` | Native gateway and sidecar. |
| `%LOCALAPPDATA%\Programs\DefenseClaw\bin\defenseclaw-hook.exe` | No-console native hook launcher. |
| `%LOCALAPPDATA%\Programs\DefenseClaw\runtime\python\` | Embedded managed Python runtime and site packages. |
| `%LOCALAPPDATA%\DefenseClaw\InstallerCache\DefenseClawSetup-x64.exe` | Maintenance copy used for repair, uninstall, and installer-based upgrades. |
| `%USERPROFILE%\.defenseclaw\config.yaml` | Primary configuration. |
| `%USERPROFILE%\.defenseclaw\audit.db` | Audit history and policy state. |
| `%USERPROFILE%\.defenseclaw\gateway.jsonl` | JSONL runtime events. |

Setup uses the Windows profile Known Folder for `%USERPROFILE%\.defenseclaw`
and does not accept an environment-controlled install or data path. Executables remain in the product command directory so
[`defenseclaw upgrade`](/docs/get-started/upgrade) can hand off file replacement
to the installer transaction.

## Safe manual testing

Use a disposable Windows user account or VM when testing installer builds. The
installer writes HKCU Installed Apps entries and updates that user's `PATH`;
testing in a throwaway profile avoids overwriting an active DefenseClaw
installation. Unsigned local artifacts should be labeled as test builds and
never treated as production releases.

## Operations and limits

```powershell
defenseclaw-gateway restart
defenseclaw-gateway stop
defenseclaw upgrade
defenseclaw doctor
```

Native Windows is hook-only. Sandbox, enterprise hooks, OpenHands, OmniGent,
OpenClaw, ZeptoClaw, and native desktop toasts are unsupported on Windows.
Bundled Local Splunk and the separate local observability stack are supported
from PowerShell/cmd on native Windows x64 with Docker Desktop using Linux
containers through the Hyper-V backend on Windows Pro, Enterprise, or
Education. Docker Desktop per-user and WSL-only installations are outside this
no-WSL certification. Both controllers invoke `docker.exe compose` directly;
they do not require Bash, WSL, MSYS, Git Bash, `.cmd`, or `.bat` lifecycle
shims. Remote observability sinks remain available independently. Inventory,
scanners, audit storage, alerts, and the TUI remain available. The TUI accepts
Windows arrow-key sequences and falls back to numbered prompts when a console
cannot redraw safely.

```powershell
defenseclaw setup splunk `
  --logs `
  --accept-splunk-license `
  --non-interactive

# Splunk Free mode does not require a Web login. Reveal the generated HEC and
# runtime-only bootstrap secrets only when explicitly needed:
defenseclaw setup splunk --show-credentials

# Stop only the owned project; named volumes and indexed data remain:
defenseclaw setup splunk --disable --logs
```

## Troubleshooting

<Accordions>
  <Accordion title="defenseclaw is not recognized">
    Open a new PowerShell or cmd window. To repair only the current session,
    run `$env:Path = "$env:LOCALAPPDATA\Programs\DefenseClaw\bin;$env:Path"`,
    then `Get-Command defenseclaw`.
  </Accordion>
  <Accordion title="Windows reports an unknown publisher">
    Local and pull-request artifacts may be unsigned. Production release
    artifacts must validate as Authenticode signed by the DefenseClaw publisher
    and match the release checksum before installation or upgrade.
  </Accordion>
  <Accordion title="The gateway runs but hooks do not fire">
    Run `defenseclaw-gateway restart`, `defenseclaw doctor`, and `defenseclaw
    status`. Confirm the agent and DefenseClaw run as the same Windows user,
    re-run `defenseclaw setup <name>`, and restart the agent.
  </Accordion>
  <Accordion title="A connector is missing from the picker">
    Confirm it is installed natively for the same Windows user and run
    `defenseclaw doctor`. Unsupported and WSL-only connectors are intentionally
    hidden. Other connectors require separate certification.
  </Accordion>
  <Accordion title="Scanner commands are reported missing">
    Re-run `defenseclaw init`, then `defenseclaw doctor`. DefenseClaw checks the
    managed runtime's command directory before the system `PATH`, so the bundled
    scanners do not need a separate global installation.
  </Accordion>
</Accordions>
