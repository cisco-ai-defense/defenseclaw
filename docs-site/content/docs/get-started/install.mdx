DefenseClaw ships pre-built release binaries for macOS and Linux, plus signed
native x64 Windows Setup in `0.8.6`. The macOS/Linux installer drops the
`defenseclaw` CLI and `defenseclaw-gateway` sidecar into `~/.local/bin/`, creates
`~/.defenseclaw/`, and sets up a virtualenv under `~/.defenseclaw/.venv/`.

<Callout title="0.8.5 was POSIX-only">
Release `0.8.5` published no Windows gateway, rollback, SBOM, or Setup artifact,
so it is not a Windows upgrade baseline. Do not copy its POSIX artifacts onto
Windows. Release `0.8.6` supplies Windows through the signed native Setup
artifact below.
</Callout>

<Callout title="Signed native Windows Setup in 0.8.6">
`DefenseClawSetup-x64.exe` provides a graphical wizard and silent mode for the
CLI/TUI, native gateway, no-console hook launcher, and an embedded managed
Python runtime. Application files live under
`%LOCALAPPDATA%\Programs\DefenseClaw`; user state remains under
`%USERPROFILE%\.defenseclaw`.
</Callout>

<Callout title="POSIX install scripts are for fresh hosts only">
The macOS/Linux release scripts stop without changing the host when they detect
an existing DefenseClaw CLI, gateway, or managed virtual environment. Use the
release-owned [upgrade path](/docs/get-started/upgrade) for an existing POSIX
installation; it enforces the signed manifest, any required bridge release,
rollback preparation, migrations, and post-upgrade health checks. Native
Windows Setup separately supports repair and same-version servicing.
</Callout>

<Callout>
Installing as `root` is supported but discouraged — DefenseClaw is per-user by design so each operator's audit DB and connector state stays isolated.
</Callout>

## Prerequisites

| Requirement | Version |
| --- | --- |
| Python | 3.10+ for source builds |
| Go | 1.26.4+ (only for source builds) |
| Node.js | 18+ (only for the OpenClaw plugin, source builds) |
| Docker | Optional, for local observability and Splunk bundles |

Cosign is not a macOS/Linux prerequisite for the `0.8.5+` release installer.
If no compatible `cosign` exists on `PATH`, the installer downloads pinned
Cosign `2.6.3` into its owner-only staging directory, validates a hard-coded
platform SHA-256 before executing it, verifies the DefenseClaw release, and
retires the temporary verifier. Nothing is installed globally.

The `0.8.6` native Windows Setup bundles its managed runtime. A packaged Windows
install does not require Python, `uv`, Go, Node.js, Git, or a
PowerShell installation command after the signed Setup artifact is downloaded.

## What's bundled (LLM backends)

The base install ships everything DefenseClaw needs to talk to the major LLM backends out of the box:

| Backend | Status | Notes |
| --- | --- | --- |
| OpenAI / Anthropic / generic OpenAI-compatible | bundled | LiteLLM, the default LLM transport, is a base dependency. |
| AWS Bedrock | bundled | `boto3` ships in `[project].dependencies` so SigV4, IAM credential modes, and Anthropic-on-Bedrock bearer tokens work without a follow-up `pip install`. |
| Google Vertex AI | **opt-in extra** | The Vertex SDK pulls ~286 MB of transitive deps (BigQuery, Cloud Storage, gRPC, protobuf). The `[vertex]` project extra is available to isolated source builds; do not package-install DefenseClaw over a release-managed runtime to add it. The gateway falls back to LiteLLM's REST transport when the SDK is absent, so basic Vertex calls still work — only service-account / workload-identity auth needs the SDK. |
| Azure OpenAI | bundled | Routed via LiteLLM's `azure_openai` adapter. |
| Self-hosted / internal endpoints | bundled | Configure via `defenseclaw setup provider add` → see [Unified LLM key → Custom providers](/docs/setup/unified-llm-key#custom-providers-internal-self-hosted-endpoints). |

## Pick an install path

<Tabs items={['macOS/Linux curl', 'Windows Setup (0.8.6)', 'From source']}>

<Tab value="macOS/Linux curl">

```bash
VERSION=0.8.6
INSTALL_URL="https://raw.githubusercontent.com/cisco-ai-defense/defenseclaw/${VERSION}/scripts/install.sh"
curl -LsSf "$INSTALL_URL" | VERSION="$VERSION" bash
defenseclaw init
```

The installer fetches the selected release for your platform and lays down a
new installation; it does not configure DefenseClaw. The follow-up
[`defenseclaw init`](/docs/get-started/quickstart) configures DefenseClaw
interactively. The installer will not replace an existing installation. Follow
the [`upgrade` guide](/docs/get-started/upgrade) instead: a `0.8.3`-or-older
host must run the authenticated current release-owned resolver so the resolver
can hop through the `0.8.4` bridge; the host's already-published built-in
upgrade command cannot perform that handoff retroactively.

If you'd rather skip the wizard entirely (CI, scripted demos), use the zero-prompt sibling instead:

```bash
defenseclaw quickstart --connector codex
```

</Tab>

<Tab value="Windows Setup (0.8.6)">

<Callout title="0.8.6 release artifact">
Download `DefenseClawSetup-x64.exe` from the `0.8.6` GitHub release. Release
`0.8.5` had no Windows Setup artifact and remains outside the Windows
upgrade-baseline matrix; source builds are developer tooling, not a substitute
for a release upgrade or rollback contract.
</Callout>

```powershell
.\DefenseClawSetup-x64.exe
```

Starting with release `0.8.6`, double-click the authenticated artifact for the
graphical setup wizard. It installs the CLI/TUI, gateway,
hook launcher, and managed runtime for the current Windows user. The finish
screen will point to `defenseclaw init` and `defenseclaw tui`; Setup does not
install a separate native DefenseClaw GUI application.

For non-interactive installs, pass properties directly:

```powershell
.\DefenseClawSetup-x64.exe /quiet /norestart INSTALLSCOPE=user CONNECTOR=codex MODE=observe STARTGATEWAY=1
```

Use `CONNECTOR=claudecode` for Claude Code or `CONNECTOR=none` to configure
later. The setup executable contains the offline payload required for install;
after download it does not require Python, `uv`, Go, Node.js, Git, or a
PowerShell installation command.

Native Windows uses agent hooks rather than the OpenClaw/ZeptoClaw proxy path.
See [Native Windows](/docs/get-started/windows) for the connector
matrix, repair, servicing, uninstall, file locations, and troubleshooting.

</Tab>

<Tab value="From source">

| Goal | Command |
|------|---------|
| Normal same-checkout development | `make all` |
| Build artifacts without installing | `make build` |
| Show developer workflow help | `make help` |
| Upgrade a packaged release | `defenseclaw upgrade` |

```bash
git clone https://github.com/cisco-ai-defense/defenseclaw.git
cd defenseclaw
make all
defenseclaw init
```

`make all` installs the editable Python CLI, builds and installs the Go gateway,
updates your PATH, and runs quickstart. The OpenClaw plugin is installed when
`CONNECTOR=openclaw` is selected. Then `defenseclaw init` walks you through
first-run setup (or run `defenseclaw quickstart --connector <x>` for the
zero-prompt path).

Source builds are development tooling, not a packaged upgrade path. `make all`
is the explicit developer reinstall workflow: it may reclaim markerless or
older source state only when the installed CLI belongs to this exact checkout.
It validates marker ownership, rejects foreign and newer source identities,
and records the current strict marker after a successful rebuild. Direct
`make install`, `scripts/install-dev.sh`, and release-managed hosts remain
fail-closed. Release installations must use `defenseclaw upgrade`; do not aim a
raw `pip`, `uv`, or editable install at a release-managed virtual environment.
The lower-level install targets are for fresh or isolated development homes,
not the normal repeated-development path.

</Tab>

</Tabs>

<Files>
  <Folder name="~/.defenseclaw" defaultOpen>
    <Folder name=".venv">
      <File name="…" />

    <File name="config.yaml" />
    <File name="audit.db" />
    <Folder name="hooks">
      <File name="codex-hook.sh" />
      <File name="claudecode-hook.sh" />
      <File name="cursor-hook.sh" />
      <File name="…" />
    </Folder>
    <Folder name="policies">
      <Folder name="guardrail">
        <Folder name="default" />
        <Folder name="strict" />
        <Folder name="permissive" />
      </Folder>
    </Folder>
  </Folder>
  <Folder name="~/.local/bin">
    <File name="defenseclaw" />
    <File name="defenseclaw-gateway" />
  </Folder>
</Files>

Release `0.8.5` has no release-managed Windows tree. Developer/source installs
use `%USERPROFILE%\.defenseclaw\` for config, audit state, hooks, and the
virtualenv, and `%USERPROFILE%\.local\bin\` for `defenseclaw.cmd` and
`defenseclaw-gateway.exe`.

With native Setup in `0.8.6`, application files live
under `%LOCALAPPDATA%\Programs\DefenseClaw`: `bin\defenseclaw.exe` launches the
embedded Python CLI/TUI, `bin\defenseclaw-gateway.exe` is the native gateway,
`bin\defenseclaw-hook.exe` is the no-console hook launcher, and
`runtime\python\` contains the managed Python runtime. User configuration,
audit state, hooks, and backups remain under `%USERPROFILE%\.defenseclaw\`.

## Verify

```bash
defenseclaw --version
defenseclaw doctor
```

`doctor` runs a full health check: it walks the config, pings the gateway, reports each active connector (one per-connector block on a multi-connector install), surfaces missing API keys, and prints a remediation hint for every red row.

## Uninstall

```bash
defenseclaw uninstall              # reversible — keeps ~/.defenseclaw/ and binaries
defenseclaw uninstall --all        # also delete ~/.defenseclaw/ (audit log, config, secrets)
defenseclaw uninstall --binaries   # also remove ~/.local/bin/defenseclaw{,-gateway}
defenseclaw uninstall --all --binaries --yes   # full nuke, no confirm prompt
```

The default tears down connector integrations (restores `~/.codex/config.toml`, `~/.claude/settings.json`, etc. from byte-for-byte backups) but **leaves `~/.defenseclaw/` and the installed binaries on disk** so a subsequent `defenseclaw setup guardrail` can pick up where you left off. Use `--all` and/or `--binaries` to make the removal total. `--dry-run` previews the plan.

With the `0.8.6` native package, use the setup executable for product uninstall:

```powershell
.\DefenseClawSetup-x64.exe /uninstall /quiet
.\DefenseClawSetup-x64.exe /uninstall /quiet DELETEUSERDATA=1
```

The first command preserves `%USERPROFILE%\.defenseclaw`; the second explicitly
removes user data.
