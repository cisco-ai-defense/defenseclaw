DefenseClaw ships a single command — `defenseclaw setup splunk` — with three independent pipelines you can mix and match:

<Callout type="info">Local `--logs` is supported on certified native Windows x64 through Docker Desktop's native `docker.exe`, Compose v2, Linux containers, and the Hyper-V backend. No Bash, WSL, MSYS, Git Bash, `.cmd`, or `.bat` lifecycle shim is used. macOS/Linux retain the existing bridge path.</Callout>

| Pipeline | Use case | What it does |
| --- | --- | --- |
| `--logs` | Local demo / dev | Brings up Splunk Free in Docker, installs the bundled local-mode app, sends events via HEC |
| `--enterprise` | Production SIEM | Configures the gateway's HEC sink to point at *your* Splunk; no Docker, no install |
| `--o11y` | Splunk Observability Cloud | Sends OTLP traces / metrics / logs to your realm |

You can run more than one. They write to different blocks in `~/.defenseclaw/config.yaml`, so adding one never undoes another.

## Local mode (`--logs`)

The fastest path to a working Splunk demo. One command brings up Splunk Free 9.x in Docker, installs the bundled `defenseclaw_local_mode` app (with five dashboards), and points the gateway's HEC sink at it.

```bash
defenseclaw setup splunk --logs --accept-splunk-license
```

What happens:

<Steps>
  <Step>
    **License check.** Without `--accept-splunk-license` the CLI prompts you interactively. Use `--non-interactive` for CI.
  </Step>
  <Step>
    **Docker and ownership preflight.** Verifies Compose v2, daemon reachability, Linux-container mode, the complete bundle, and ports `8000` (Splunk Web) and `8088` (HEC). Ports already published by the exact owned project are accepted; foreign listeners are rejected.
  </Step>
  <Step>
    **Native/bridge bootstrap.** Windows packages the app with Python and launches `docker.exe compose` using argument arrays. macOS/Linux use `splunk-claw-bridge`. A cryptographically random HEC token and runtime-required administrator bootstrap secret are stored in the protected private env file.
  </Step>
  <Step>
    **Readiness and sink wiring.** Waits for both Web and authenticated HEC health, then writes the owned `local-splunk` audit sink pointing at `https://127.0.0.1:8088/services/collector/event`, index `defenseclaw_local`, sourcetype `defenseclaw:json`, and reloads/restarts the gateway. A failure rolls configuration and runtime back.
  </Step>
  <Step>
    **App install.** Copies `bundles/splunk_local_bridge/splunk/apps/defenseclaw_local_mode` into the container — five dashboards plus shared SPL macros.
  </Step>
</Steps>

Open Splunk at [http://localhost:8000](http://localhost:8000). Splunk Free mode disables Web authentication, so there is no login to advertise. `defenseclaw setup splunk --show-credentials` explicitly reveals the HEC token and labels the administrator bootstrap secret as runtime-only. The bundled app shows up as **DefenseClaw (local mode)**.

Five dashboards land under **Demo Dashboards** in the app navigation. They're the same ones you'd build by hand against a real production deployment, just pointed at the local index for a self-contained demo.

### What the dashboards query

All five views read from a shared set of SPL macros under `bundles/splunk_local_bridge/splunk/apps/defenseclaw_local_mode/default/macros.conf`, scoped to `index=defenseclaw_local`. The macros normalize the gateway's JSON envelope so the dashboards don't have to repeat field extractions.

The gateway ships audit events: the discriminator is the top-level `action` field (the `audit.Action` enum — `connector-hook`, `guardrail-block`, `guardrail-verdict`, `approval-request`, …), **not** a `kind` field. The connector dimension lives in the top-level `connector` field and, on hook rows, in `structured.connector`.

Useful starting queries:

```spl title="Codex hook activity in the last hour"
index=defenseclaw_local sourcetype=defenseclaw:json action="connector-hook" structured.connector="codex"
| stats count by structured.event_type, structured.action, structured.would_block
| sort -count
```

```spl title="Blocked guardrail actions, broken down by connector"
index=defenseclaw_local sourcetype=defenseclaw:json action="guardrail-block"
| stats count by connector, structured.rule_id, structured.severity
| sort -count
```

```spl title="Hook throughput per connector (spot a noisy agent)"
index=defenseclaw_local sourcetype=defenseclaw:json action="connector-hook"
| timechart span=5m count by structured.connector
```

## Enterprise mode (`--enterprise`)

Send events to an existing Splunk deployment. No Docker, no app install — just configure the HEC sink.

```bash
defenseclaw setup splunk --enterprise \
  --hec-endpoint https://splunk.example.com:8088 \
  --hec-token "$DEFENSECLAW_SPLUNK_HEC_TOKEN" \
  --index defenseclaw \
  --sourcetype _json
```

| Flag | Purpose |
| --- | --- |
| `--hec-endpoint` | Full HEC URL ending in `/services/collector/event`. The CLI accepts both bare host and full URL. |
| `--hec-token` | HEC token. Reads `DEFENSECLAW_SPLUNK_HEC_TOKEN` if omitted. |
| `--index` | Defaults to `defenseclaw`. Create it ahead of time. |
| `--sourcetype` | Defaults to `_json`. Use `defenseclaw:json` if you've imported the bundled macros. |
| `--app-name` | Pretty name shown in the gateway TUI as the sink label. |
| `--skip-test` | Skip the live HEC probe (useful in air-gapped CI). |

The CLI runs a one-shot probe by default — it sends a test event and confirms HEC returns 200 before persisting the config. If your HEC is behind a VPN, pass `--skip-test`.

### Field mapping in your Splunk

The events arrive as JSON with the envelope described in [Observability](/docs/observability). For Splunk Cloud or Splunk Enterprise without the bundled app, you'll typically want:

```ini title="props.conf"
[defenseclaw:json]
KV_MODE = json
TIME_PREFIX = "ts":"
TIME_FORMAT = %Y-%m-%dT%H:%M:%S.%QZ
SHOULD_LINEMERGE = false

LOOKUP-decision = decision_lookup decision
EXTRACT-trace = trace_id="(?<trace_id>[0-9a-f]+)"
```

We don't ship a transforms-only TA today; the local-mode app stays Splunk-Free-friendly so you can copy fields out as needed.

## Splunk Observability Cloud (`--o11y`)

For organizations standardised on Splunk O11y instead of Splunk Enterprise, the gateway's OTel exporter sends to the OTLP HTTP ingest endpoint:

```bash
defenseclaw setup splunk --o11y \
  --realm us1 \
  --access-token "$SPLUNK_ACCESS_TOKEN" \
  --traces --metrics --logs
```

This writes the `otel:` block in `config.yaml` (same as `setup local-observability`, but pointed at `https://ingest.us1.signalfx.com`). Toggle individual signals with `--no-traces` / `--no-metrics` / `--no-logs` if you don't pay for all three.

## Disable

```bash
defenseclaw setup splunk --disable --logs
defenseclaw setup splunk --disable --enterprise
defenseclaw setup splunk --disable --o11y
```

`--disable` disables the matching destination (and stops the exact owned local project if you ran `--logs`). Local named volumes are preserved, and operator-authored loopback or remote HEC sinks are not touched.

## Verify

```bash
defenseclaw doctor

tail -f ~/.defenseclaw/gateway.jsonl | jq 'select(.kind == "guardrail.verdict")'
```

`doctor` confirms the HEC endpoint is reachable and the token is valid. The JSONL tail mirrors what's being shipped to Splunk (the gateway writes the same payload to local JSONL and remote sinks) — useful when you suspect a sink is dropping events.

## Troubleshooting

<Accordions>
  <Accordion title="Local Splunk container won't start (port 8000 in use)">
    Stop the conflicting service or run `defenseclaw setup splunk --disable --logs` and retry. There's no `--port` flag today. A listener is accepted only when Docker proves that the exact DefenseClaw-owned Compose project publishes the expected loopback mapping.
  </Accordion>
  <Accordion title="Enterprise HEC returns 403">
    Token is wrong or HEC isn't enabled on that Splunk endpoint. Test with curl: `curl -k $HEC_ENDPOINT -H "Authorization: Splunk $TOKEN" -d '{"event":"test"}'`. The CLI's `--skip-test` skips this same probe.
  </Accordion>
  <Accordion title="Events arriving but no extracted fields">
    Make sure your sourcetype is `_json` (auto-extract) or that you've installed the bundled macros / props.conf snippet above for `defenseclaw:json`.
  </Accordion>
  <Accordion title="O11y traces but no logs">
    Splunk O11y log ingest is a separate entitlement. The exporter sends to the same OTLP endpoint but the realm may not accept logs — check with `--logs` toggled off if you only have traces/metrics.
  </Accordion>
</Accordions>
