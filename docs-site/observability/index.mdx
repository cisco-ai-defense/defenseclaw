## Overview

Observability in DefenseClaw has three layers, emitted in parallel:

1. **Audit store** — authoritative, persisted SQLite database of every decision, verdict, and action.
2. **JSONL local log** — append-only structured events on disk, readable by Splunk Universal Forwarder and `jq`.
3. **OTel + audit sinks** — OpenTelemetry traces/metrics plus audit-event delivery to Splunk HEC, OTLP logs, or HTTP JSONL.

The streams share correlation and provenance fields, but they are not identical copies: the audit bridge translates persisted audit rows into lifecycle JSONL events, while native guardrail/judge/scan emitters write richer structured rows directly.

## Which signal to use

| Question | Start here | Why |
|----------|------------|-----|
| Did DefenseClaw make a decision? | [Audit store](/docs-site/observability/audit-store) | SQLite is the persisted source of record. |
| Did an event leave the process? | [Gateway JSONL log](/docs-site/observability/gatewaylog-jsonl) | JSONL shows what the local writer emitted. |
| Did the SIEM receive it? | [Sinks](/docs-site/observability/sinks) | Sink probes isolate delivery and auth issues. |
| Did a webhook receiver reject it? | [Webhook dispatcher](/docs-site/observability/webhook-dispatcher) | Dispatcher docs cover retry, HMAC, cooldown, and SSRF validation. |
| Did PII get handled correctly? | [Redaction](/docs-site/observability/redaction) | Redaction behavior differs by display and sink path. |

## First checks

```bash
defenseclaw-gateway status
defenseclaw setup observability list
defenseclaw-gateway audit export --limit 5
```

If the audit export is empty, debug the producing subsystem first. If the export has rows but the external destination is empty, debug the configured sink or webhook.
