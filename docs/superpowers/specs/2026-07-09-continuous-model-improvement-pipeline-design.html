1. Executive Summary

1.1 Purpose

DefenseClaw manages an automated pipeline that continuously improves local LLM models by learning from production traffic. User queries are classified by the semantic router, stored as training data, used to fine-tune local models, and promoted to serve traffic when they match frontier model quality — all orchestrated by DefenseClaw with zero manual intervention after initial setup.

1.2 Goals

#GoalMetric
G1Replace expensive frontier API calls with local modelsCost reduction per category
G2Maintain quality parity with frontier≥90% of frontier score (LLM-as-judge)
G3Zero manual intervention after setupAuto-trigger, train, promote, rollback
G4Single source of truthAll config in ~/.defenseclaw/config.yaml
G5Graceful degradationPipeline failure never impacts user requests

1.3 Key Decisions

Training triggerAuto (threshold) + manual CLIHands-off by default, override when needed
Trace storageTempo (viz) + SQLite (training)Best of both: dashboards + fast extraction
Training backendUser selects: Unsloth or mlx-lm-loraUnsloth for CUDA GPUs, mlx-lm-lora for Mac
AlgorithmUser selects per categoryDifferent categories need different approaches
Base modelsCurated list (Qwen, Llama, Phi, Gemma)Tested + reliable
Model hostingllama-server, always running, lazy-loadInstant inference on promotion
EvaluationLLM-as-judgeMost accurate for open-ended quality
PromotionAutomatic if ratio ≥ thresholdRemoves human bottleneck
RollbackAuto (quality drop) + manual CLISafety net for production

3. Data Flow Diagrams

3.1 Runtime Data Flow (Every Request)

  REQUEST PATH (synchronous, <200ms total)

  User ─────▶ DefenseClaw Gateway
              │
              ├─▶ [1] Pre-call Guardrails (regex + judge + policy)
              │       IF blocked → return error
              │       IF passed ↓
              │
              ├─▶ [2] SR classify/intent (POST :8888/api/v1/classify/intent)
              │       Returns: {category, recommended_model, matched_signals}
              │
              ├─▶ [3] Model Selection (Bifrost)
              │       IF category promoted → llama-server (:8090)
              │       ELSE → frontier API (Anthropic/OpenAI/Bedrock)
              │
              ├─▶ [4] Post-call Guardrails (response inspection)
              │
              └─▶ [5] Return response to user

  ───────────────────────────────────────────────────────────

  TRACE PATH (asynchronous, non-blocking, after step 5)

  ├─▶ [A] Write to training-store.db (SQLite)
  │       {category, prompt, response, model_used, latency, tokens}
  │
  └─▶ [B] Export OTel span to Tempo
          {dc.routing.category, dc.routing.decision, ...}

4. Component Specifications

4.1 Trace Capture

Purpose: Store every LLM request + SR classification for future training.

Implementation: Buffered channel (100 entries) + single writer goroutine. Never blocks response path.

Design choice: SQLite for training extraction (fast, indexed, no network). Tempo for visualization (dashboards, TraceQL queries). Both written in parallel.

SQLite Schema

CREATE TABLE training_traces (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    timestamp TEXT NOT NULL DEFAULT (datetime('now')),
    category TEXT NOT NULL,
    recommended_model TEXT,
    matched_signals TEXT,
    prompt TEXT NOT NULL,
    response TEXT NOT NULL,
    model_used TEXT NOT NULL,
    is_promoted_model BOOLEAN DEFAULT FALSE,
    latency_ms INTEGER,
    tokens_in INTEGER,
    tokens_out INTEGER,
    used_for_training BOOLEAN DEFAULT FALSE,
    training_run_id TEXT
);

CREATE INDEX idx_category_unused ON training_traces(category, used_for_training);
CREATE INDEX idx_category_time ON training_traces(category, timestamp);

4.2 Auto-Trigger

Mechanism: Single goroutine, checks trace counts per category every 60 seconds. Fires pipeline when count ≥ min_traces and no training in progress.

4.3 Dataset Builder

BackendMethodOutput
unslothdata-designer Python API (LLM generation + judging)Rich DPO/SFT dataset with quality scores
mlx-lm-loraDirect JSONL conversion (Go code)Simple format mapping

4.4 Training Runner

Generates a Python training script, executes as subprocess, monitors stdout for progress, handles 12-hour timeout.

4.5 Evaluator (LLM-as-Judge)

For each held-out prompt: get local response, look up frontier response, send both to judge model with scoring rubric. Compute ratio.

Judge Prompt:
"Rate Response A vs B on accuracy, helpfulness, completeness (1-10).
Return JSON: {"score_a": N, "score_b": N}"

Scoring:
  ratio = avg(score_a across 50 prompts) / avg(score_b)
  IF ratio ≥ 0.90 → PROMOTE
