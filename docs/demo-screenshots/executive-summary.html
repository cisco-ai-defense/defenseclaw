1. Executive Overview

DefenseClaw now provides capability-based access control for AI agents — a granular, runtime-enforced system that goes far beyond binary block/allow decisions. Instead of giving an agent full access to an MCP server or none at all, capabilities define exactly what each agent can do, with constraints, restrictions, time windows, and rate limits.

The system is built in two phases: Phase 1 delivers the evaluator engine, policy format, CLI commands, TUI integration, and audit logging. Phase 2 adds automatic policy generation that introspects MCP/skill manifests and adapts security posture based on scan findings.

3. Phase 1: Capability Evaluator

Policy Manifest Format

Each agent gets a .capability.yaml file in ~/.defenseclaw/capabilities/. Policies express what an agent can do with fine-grained constraints:

support-bot.capability.yaml
agent: support-bot description: "Customer support automation agent" capabilities: - name: read_jira_ticket resource: "jira.get_issue" constraints: project: "ENG-*" # glob pattern fields: [summary, status, assignee] # list membership - name: post_slack_message resource: "slack.post_message" constraints: channel: "#support" template: approved_templates_only - name: read_confluence resource: "confluence.get_page" constraints: space: SUPPORT restrictions: - "no_external_http" - "no_bulk_export" conditions: time_window: "09:00-18:00" # UTC environments: [production, staging] rate_limit: max_calls: 100 window_seconds: 3600

Constraint Matching & Restrictions

String Values — Glob Patterns

Uses filepath.Match for glob matching. project: "ENG-*" allows ENG-123 but denies FIN-456.

Slice Values — List Membership

All requested fields must be in the allowed list. fields: [summary, status] allows requesting summary but denies comments.

Restrictions — Hard Blocks

Checked before capabilities. No capability can override a restriction.

RestrictionBlocks
no_external_httphttp.*, external_http.*
no_bulk_export*.export_all, *.bulk_export
no_write*.create_*, *.update_*, *.post_*
no_delete*.delete_*, *.remove_*

Conditions (Agent-Global)

Time Window

UTC-based, supports midnight crossing (e.g., 22:00-06:00). Calls outside the window are denied.

Environment

Restricts agent to named environments. Empty list allows all.

Rate Limiting

Sliding window tracked in SQLite capability_calls table. Exceeding the limit → deny until window clears.

Default-Deny

Unknown agents, unmatched resources, and failed constraints all result in denial.

Key principle: Capabilities are not tools — they are constrained permissions. An agent may have access to jira.get_issue but only for project ENG-* during business hours with a 100-call/hour limit.

Audit Trail

Every evaluation decision is logged to the SQLite audit store with timestamp, agent, resource, parameters, allowed/denied, and reason. Rate limit calls are tracked in a separate capability_calls table with indexed queries for efficient sliding-window calculations.

5 CLI commands: list, show, evaluate, validate, and approve (Phase 2). All follow existing Cobra patterns. Decisions are also forwarded to Splunk HEC and OTel.

5. Phase 2: Scan-Driven Auto-Generation

Without auto-generation, users must manually write .capability.yaml files for every agent. Phase 2 solves this by introspecting manifests for per-tool granularity and adapting security posture based on scan findings.

MCP Manifest Introspection

Parses the MCP server's JSON manifest to discover per-tool metadata:

Terminal — MCP Introspection
=== Step 1: MCP Manifest Introspection === Parsed 3 tools from alert-service MCP manifest: [tool] get_weather Get current weather for a location [tool] create_alert Create a new alert rule [tool] delete_alert Delete an existing alert

Skill Manifest Introspection

Terminal — Skill Introspection

Skill Policy with read-only Permission

Terminal — Skill Policy Generation
=== Skill Policy (read-only permissions) === Agent: auto-permissioned-skill Capabilities: 1 permissioned-skill -> permissioned-skill.* Restrictions: [no_write no_delete]
Read-like detection: Tools with prefixes get_, list_, read_, search_, fetch_, query_, describe_, show_ are considered safe for the restrictive posture. All other tools are filtered out when HIGH/CRITICAL findings exist.

Watcher Integration

Auto-generation hooks directly into runAdmission() at 5 return points (allowed/clean/warning verdicts). No new goroutines or services — it runs inline after the admission verdict.

Conflict resolution:
  • Manual policy exists → skip auto-generation entirely (manual always wins)
  • Auto policy already exists → skip (don't overwrite on re-scan)
  • Introspection fails → fall back to wildcard <name>.* capability
  • Approve when manual exists → error (user should edit manual file)

Key Design Decisions:
  • Default-deny — unknown agents and unmatched resources are always rejected
  • Sidecar/hook mode — no MCP proxy required, integrates with existing daemon
  • Scan-informed posture — security adapts automatically to scan findings
  • No circular imports — GenerateRequest uses primitive types + ScanResultSummary
  • Semi-blocking approval — policies active immediately, flagged pending until approved
  • Graceful degradation — introspection failures fall back to wildcard capabilities
