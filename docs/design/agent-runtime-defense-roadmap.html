Executive Positioning

Current DefenseClaw

DefenseClaw already inspects agent intent through hooks, guardrail rules, OPA/Rego verdicts, scanner findings, audit, OpenTelemetry, Splunk, and TUI surfaces.

It answers whether the prompt, tool call, command text, file path argument, MCP server, or provider request should be allowed.

Next Runtime Layer

The new layer enforces what the workstation actually does: process launches, child processes, file opens, file writes, network flows, and credential use.

This creates defense in depth for Codex, Claude Code, Cursor, Windsurf, Gemini CLI, Copilot CLI, and future coding agents.

Product thesis: hooks remain the semantic layer, OS runtime controls become the truth layer, and credential injection becomes the high-assurance managed-run layer.

Core Upgrade

Hooks Answer

What did the agent say it wanted to do?

Hooks are high-signal because they see intent: prompt text, tool names, command strings, MCP metadata, file arguments, and model/provider context.

OS Runtime Controls Answer

What did the machine actually do?

Runtime controls observe and block the concrete operating-system behavior, even when the agent uses a subprocess or native API that bypasses the hook payload.

Defense-in-Depth Architecture

DefenseClaw should support both already-installed/running agents and managed-run sessions. Transparent protection gives broad enterprise coverage; managed run gives the strongest sandboxing and credential guarantees.

Policy and Enforcement Flow

Full OPA/Rego policy should not run inside kernel callbacks, System Extension callbacks, or driver hot paths. The gateway evaluates rich policy, compiles compact runtime profiles, and OS providers enforce locally.

Runtime Event Enforced By Policy Inputs Typical Verdict
runtime.file_open macOS EndpointSecurity, Windows minifilter, Linux eBPF LSM path, operation, process tree, workspace, user, sensitivity Deny reads of SSH keys, cloud credentials, .env, browser profiles
runtime.exec macOS EndpointSecurity, Windows process controls, Linux eBPF LSM/seccomp binary path, signer, argv, parent process, connector, managed mode Observe or block suspicious exec and known exfiltration helpers
runtime.network_flow macOS NetworkExtension, Windows WFP, Linux cgroup/tc eBPF host, IP, port, protocol, process identity, route policy Block metadata endpoints, non-allowlisted egress, provider bypass
runtime.credential_use Credential gateway and policy service Duo user, group, device, repository, API action, destination Inject, deny, or require approval/step-up

Credential Injection Flow

Credential injection requires a network path DefenseClaw controls. The agent receives only a placeholder. The gateway injects the real credential after identity, destination, and action policy allow the request.

Service / Protocol Policy Facts Injection Slot Example Control
HTTPS / REST host, method, path, headers, body metadata Authorization header, cookie, custom header, mTLS identity Allow GET, require approval for DELETE, deny unknown hosts
GitHub org, repo, endpoint, action, token scope Bearer token or Git credential helper path Map Duo group to org/repo credential binding
Kubernetes verb, resource, namespace, object name API bearer token or client certificate Deny secrets; step-up for prod deploys
Postgres database, user, statement type, table names startup/auth handshake Read-only for dev; approval for prod mutation
SSH command, subsystem, forwarding mode, destination gateway-owned upstream key or certificate Block port forwarding; log exec commands

Duo Identity and Credential Mapping

Duo can supply the user identity and authentication assurance needed to decide whether a managed agent session may receive a credential at the gateway.

Duo Signal DefenseClaw Mapping Credential Gateway Use Splunk Field Examples
SSO OIDC/SAML subject, username, email Runtime user identity and agent session owner Associate managed-run session with human user user.name, user.email, duo.subject
Duo group membership Credential binding and policy scope Map group to GitHub org, Kubernetes namespace, DB role duo.groups, credential.binding_id
Duo devices / phones / hardware tokens Known user authentication factors and device context Require step-up for sensitive credential injection duo.device_id, auth.factor
Duo authentication logs Recent MFA outcome and assurance evidence Correlate credential injection with strong authentication duo.auth.result, duo.auth.time
DefenseClaw device identity Gateway-side Ed25519 device fingerprint and endpoint context Bind user + device + agent session before injection defenseclaw.device.fingerprint, host.name
Recommended model: use Duo SSO or Universal Prompt for interactive authentication and step-up, use the Duo Admin API to sync users/groups/devices/logs into a DefenseClaw identity map, then use that identity map in credential binding policy.

Splunk and Observability Sync

DefenseClaw already has Splunk HEC support, OpenTelemetry schemas, runtime alert schemas, and local Splunk dashboards. The runtime-control roadmap should extend those same channels instead of creating a separate event stream.

Signal Splunk Path Why It Matters Example Fields
Audit and policy events HTTP Event Collector Splunk HEC supports sending application events over HTTP/HTTPS with token authentication. event.type, severity, connector, policy.rule_id
Metrics, traces, logs Splunk Distribution of the OpenTelemetry Collector Collector path supports standard logs, metrics, and traces into Splunk Observability Cloud. trace_id, span_id, service.name, deployment.environment
Runtime control events HEC + OTel logs Gives SOC and engineering a single view across prompt intent and OS behavior. runtime.action, runtime.resource.path, runtime.destination.host
Credential gateway events HEC + OTel spans Shows when credentials were denied, injected, or required approval. credential.id, credential.scope, duo.user_id, approval.status
Duo identity evidence Duo Admin API ingest + joined lookup Correlates runtime action to authenticated user, group, and MFA context. duo.user_id, duo.groups, duo.auth.result
Dashboard outcomes: runtime deny trends, top bypass-suspected agents, sensitive file access attempts, non-allowlisted egress, credential injection by Duo group, approval latency, and policy verdict distribution by connector and platform.

Policy Position

Existing agents continue to work during observe mode. Production credentials require managed run or successful migration to placeholder credentials. Deterministic HIGH and CRITICAL runtime events can move to action mode first.

Stage Goal Default Mode
Pilot Validate discovery, event volume, policy mapping, false positives, Splunk field quality. Observe
Fleet Observe Deploy OS provider broadly and collect runtime evidence across agents and platforms. Observe
Deterministic Blocks Block HIGH/CRITICAL sensitive file, suspicious exec, and egress rules. Action for deterministic rules
Managed Credential Access Require managed run for production GitHub, Kubernetes, cloud, DB, and SSH credentials. Action
Full Defense in Depth Hooks, runtime controls, credential gateway, Duo identity, Splunk dashboards, and policy correlation active. Action

Recommended Build Order

  1. Shared runtime-control schema, target registry, normalized event model, and policy compiler.
  2. Duo identity map design: user, group, device, auth logs, DefenseClaw device fingerprint.
  3. Splunk field mapping and dashboards for runtime events, credential injection, and Duo correlation.
  4. macOS EndpointSecurity observe mode for exec and sensitive file opens.
  5. macOS action mode for deterministic HIGH/CRITICAL file and exec rules.
  6. macOS NetworkExtension observe/action mode for agent egress.
  7. Managed-run launcher, shims, shortcuts, and broker tools.
  8. Credential gateway MVP for HTTPS, GitHub, and Kubernetes.
  9. Linux managed-run hardening and eBPF transparent mode.
  10. Windows WFP network controls and minifilter filesystem controls.
